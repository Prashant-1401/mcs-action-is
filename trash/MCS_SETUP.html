System Overview

Architecture

MCS Actions is a three-tier system. Google Sheets is the database, a GAS Web App is the write API, and FastAPI provides Gemini AI features. The React SPA reads data via CSV gviz URLs and writes via the GAS endpoint.

📖 Data Flow
Reads: React fetches CSV via docs.google.com/spreadsheets/d/{SHEET_ID}/gviz/tq?tqx=out:csv&sheet=TabName — no auth required, sheet must be publicly shared.

Writes: React POSTs JSON to the GAS Web App (VITE_SHEET_SCRIPT_URL). GAS validates and calls SpreadsheetApp.openById(SHEET_ID).

AI Features: Meeting transcription, action extraction, and translation are handled by the FastAPI backend at VITE_API_BASE_URL using gemini-flash-lite-latest.

Step 1

Google Sheet Configuration

The sheet is the live database. All tabs must exist with exact column headers before the app runs.

1
Create a new Google Sheet
Go to sheets.google.com → New. Name it "MCS Action Log". Copy the Sheet ID from the URL: docs.google.com/spreadsheets/d/THIS_PART/edit
2
Share the sheet publicly (for CSV reads)
File → Share → Share with others → Change to "Anyone with the link" → Viewer. This allows the React app to read data via the gviz CSV URL without authentication.
3
Run setupMissingTabs() from Apps Script
After deploying the GAS code (Step 2), run the setupMissingTabs() function once from the editor. This creates all required tabs with correct headers and seeds the Reasons tab.
4
Populate Users tab with credentials
The app authenticates exclusively from the Users sheet. Each user row must have username and password columns. There are no hardcoded fallback credentials — if the sheet is unreachable, login is blocked.
⚠
Passwords in the Users tab are stored in plain text. This is acceptable for an internal tool but avoid reusing real passwords. A future upgrade should hash passwords with SHA-256 (see Gas_Script_v2.js for the implementation).

Step 2

Apps Script Web App (GAS)

The GAS script is the write API for the sheet. It handles insert, update, delete, upsert, and replace_all operations on any registered tab.

1
Open the Apps Script editor
From the Google Sheet → Extensions → Apps Script. Paste the full contents of Code.gs (the GAS script file from your repository).
2
Verify SHEET_ID is correct
At the top of the script, confirm:
GAS
const SHEET_ID = "your-sheet-id-here";
Replace with your actual Sheet ID from Step 1.
3
Run setupMissingTabs() once
In the editor, select setupMissingTabs from the function dropdown → Run. This creates all missing tabs and adds reasonOfAction and machineName columns to the Actions tab. Check the Execution Log for confirmation messages.
4
Deploy as Web App
Deploy → New Deployment → Type: Web App → Execute as: Me → Who has access: Anyone → Deploy. Copy the web app URL — this becomes VITE_SHEET_SCRIPT_URL.
5
Test the deployment
Open the web app URL in a browser. You should see JSON:
JSON
{ "ok": false, "error": "Unknown tab: undefined" }
This error is expected and confirms the script is running. A 200 response with this error body means the GAS deploy is working.
ℹ
Re-deployment note: Every time you edit the GAS script, you must create a New Deployment (not redeploy an existing one) to get a new URL. Update VITE_SHEET_SCRIPT_URL in Vercel whenever this URL changes. The old URL continues working for 6 months.

Supported Actions (POST body)

ActionRequired FieldsDescription
inserttab, rowAppend a new row to the tab
updatetab, id, patchUpdate specific fields of a row by key
deletetab, idDelete a row by key value
upserttab, rowInsert if not found, update if found
replace_alltab, rowsRewrite entire tab (skips if rows is empty — safety guard)
Step 3

FastAPI Backend

The Python backend provides Gemini AI features: meeting transcript analysis, real-time paragraph insights, Hindi translation, and email escalation. It runs at mcs-action.onrender.com.

Local Development

bash
# Clone and enter the repo cd mcs-action/backend # Install dependencies pip install fastapi uvicorn google-generativeai python-dotenv requests # Create .env file (see Step 4 for values) cp .env.example .env # Run locally uvicorn main:app --reload --port 8000

API Endpoints

MethodEndpointDescription
GET/api/healthHealth check — returns Gemini config status
POST/api/meetings/extract-insightsFull transcript → structured action log (JSON)
POST/api/meetings/analyze-paragraphSingle paragraph → actions, decisions, risks, key points (real-time)
POST/api/translateHindi / Hinglish → English translation
POST/api/email/escalateSend escalation email for overdue actions
POST/api/email/share-insightsEmail meeting insights to recipients
ℹ
Ollama fallback: If Gemini API fails (quota exceeded, network error), the backend automatically falls back to a local Ollama instance at localhost:11434 with model llama3. Set OLLAMA_MODEL env var to change the model. On Render this fallback will not work — Gemini is the only option in production.

Gemini Model Used

🤖 gemini-flash-lite-latest
All three AI features (transcript extraction, paragraph analysis, translation) use gemini-flash-lite-latest. This is the fastest and cheapest Gemini model, appropriate for the real-time meeting analysis use case. Free tier supports 15 RPM and 1M tokens/day as of mid-2025.

Step 5

React App (Vite)

The frontend is a single-file React SPA (App.jsx) built with Vite. It uses no component library — all UI is hand-coded with inline styles.

Key Runtime Behaviour

🔐 Login
Auth reads the Users sheet tab via CSV URL. Matches username and password columns. No hardcoded fallback — sheet must be reachable.
📊 Data Load
On mount, useSheetDB fetches all 9 tabs in parallel. A 1-second delay after load prevents sync effects from overwriting the sheet with empty default arrays.
✍ Writes
State changes sync to the sheet via replace_all POST to the GAS endpoint. Empty payloads are silently skipped by the GAS safety guard.
🤖 AI
Meeting Room calls VITE_API_BASE_URL/api/meetings/analyze-paragraph in real-time and /extract-insights on session end. Translation uses /api/translate.

Step 6

Render Deployment (Backend)

The FastAPI backend is hosted on Render as a Web Service at mcs-action.onrender.com.

1
Connect GitHub repo to Render
render.com → New → Web Service → Connect Prashant-1401/mcs-action repo. Select the branch (main).
2
Configure build and start commands
render settings
Root Directory: backend Build Command: pip install -r requirements.txt Start Command: uvicorn main:app --host 0.0.0.0 --port $PORT
3
Add environment variables in Render dashboard
Dashboard → Environment → Add the 5 required env vars: GEMINI_API_KEY, SMTP_SERVER, SMTP_USER, SMTP_PORT, SMTP_PASS. Render encrypts these at rest.
4
Verify deployment
Visit https://mcs-action.onrender.com/api/health. Expected response:
JSON
{ "status": "ok", "gemini_configured": true }
⚠
Free tier cold starts: Render free tier spins down after 15 minutes of inactivity. The first request after idle takes 30–60 seconds. This causes a visible delay when opening the Meeting Room for the first time. Consider a Render Starter plan ($7/mo) or a UptimeRobot ping cron to keep it warm.
Step 7

Vercel Deployment (Frontend)

The React SPA is deployed to Vercel and served from the CDN. Zero config required beyond env vars.

1
Import the GitHub repository
vercel.com → New Project → Import Prashant-1401/mcs-action → Set Root Directory to frontend (or wherever your package.json is).
2
Add environment variables
Settings → Environment Variables → Add the 3 frontend vars: VITE_SHEET_SCRIPT_URL, VITE_SHEET_ID, VITE_API_BASE_URL. These get baked into the Vite build at deploy time.
3
Deploy
Vercel auto-detects Vite. Build command is npm run build, output is dist/. First deploy takes ~90 seconds. Every push to main triggers a redeploy automatically.
4
Update VITE_API_BASE_URL if Render URL changes
If you rename or recreate the Render service, update VITE_API_BASE_URL in Vercel env settings and redeploy. The value is baked into the JS bundle — a running deployment will not pick up env changes without a redeploy.
✅
Vercel automatically provides HTTPS, CDN distribution, and preview deployments for pull requests. No additional config needed for these.

Final Checklist

Deployment Checklist

Run through this before going live.

📊 Google Sheet
  • Sheet ID copied from URL
  • Sheet shared publicly (Anyone with link — Viewer)
  • Apps Script code deployed as Web App (Execute as: Me, Access: Anyone)
  • setupMissingTabs() run — all 11 tabs exist with headers
  • Users tab populated with at least one Admin user (username + password)
  • Plants tab populated with plant names matching app usage
  • Departments tab populated
🐍 FastAPI Backend (Render)
  • GEMINI_API_KEY set in Render environment
  • SMTP_USER and SMTP_PASS set (or left blank for demo mode)
  • GET /api/health returns {"status":"ok","gemini_configured":true}
  • CORS is set to allow_origins=["*"] — confirm this is acceptable for your security requirements
⚛ React Frontend (Vercel)
  • VITE_SHEET_SCRIPT_URL set and not the placeholder value
  • VITE_SHEET_ID matches the actual Sheet ID
  • VITE_API_BASE_URL points to the live Render URL
  • Login page shows "Connected" status (green dot) — not "Offline"
  • Admin user can log in and reach Master Setup (page 99)
  • Actions page loads real data from sheet (not empty)
  • Meeting Room can connect to AI backend (test with a short paragraph)
🔒
Security reminders before production use: (1) Remove or disable the guest account. (2) Change all default passwords. (3) Consider restricting the GAS Web App to specific users once your user base is stable (access: "Only myself" + allowlist). (4) The FastAPI CORS is currently set to allow_origins=["*"] — restrict this to your Vercel domain in production.
